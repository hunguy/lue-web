# Handoff Document

## Project Overview
The project is integrating a React-based web frontend (`lyricflow-ebook-reader`) into a Python-based terminal eBook reader application (`lue`). The goal is to provide a "read-along" web interface (similar to Apple Music's lyrics view) that synchronizes with the backend's Text-to-Speech (TTS) engine.

## Architecture
- **Backend (`lue/web.py`)**: 
  - Uses **FastAPI** to serve the web application and provide APIs.
  - **WebSocket (`/ws`)**: Provides real-time bidirectional communication. The backend streams the current sentence, timing data, and playback status to the frontend. The frontend sends commands (e.g., `seek`, `pause`, `next_sentence`).
  - **Static Serving**: The built React app (from `lyricflow-ebook-reader/dist`) is mounted and served at the root `/`.
  - **Audio Delivery**: TTS audio generated by `lue` is saved to buffer files (`buffer_0.mp3`, etc.) in the user's cache dir. The web server serves these via `/audio/{buffer_filename}`.
  - **WebLue Reader**: A subclass of the main `Lue` reader that overrides terminal UI updates and instead broadcasts state over WebSockets.

- **Frontend (`lyricflow-ebook-reader`)**:
  - React + Vite + TailwindCSS.
  - **Bookshelf (`Bookshelf.tsx`)**: Displays recently read books and allows uploading new EPUB/PDFs.
  - **Reader (`App.tsx`)**: 
    - Connects to WebSocket to receive chapter context (preloaded sentences) and live TTS updates.
    - Manages scrolling and synchronized highlighting using the `timing` data provided by the backend.
    - Uses a hidden HTML5 `<audio>` element to play the TTS buffers in sequence.

## Gotchas & Learnings (Crucial for Future Agents)

### 1. Browser Audio Caching
The backend reuses a fixed pool of buffer files (e.g., `buffer_0.mp3`, `buffer_1.mp3`) for efficiency.
* **Gotcha**: Browsers aggressively cache audio files. When the backend told the frontend to play `buffer_0.mp3` for a new sentence, the browser would play the *old* cached version of that sentence.
* **Fix**: Implemented cache-busting by appending a timestamp query parameter to the audio URL in the backend (`f"/audio/{filename}?t={time.time()}"`).

### 2. Audio Player Race Conditions during "Seek"
When a user clicks a word to seek, the frontend needs to jump to that specific sentence and word.
* **Gotcha**: If the frontend had an active `audio_url` for a sentence, clicking a word would cause the HTML5 audio element to fire `loadeddata` and consume the precise word seek-timestamp *before* the backend had finished generating the actual new TTS audio.
* **Fix**: The frontend must aggressively wipe `audio_url` from its state as soon as a `seek` is requested. This forces the audio player to wait until the backend broadcasts the new `audio_url` along with the fresh TTS payload.

### 3. Seeking to Words in Preloaded Sentences
The backend preloads the entire chapter's sentences to the frontend for free scrolling.
* **Gotcha**: Preloaded sentences have *no timing data* because TTS hasn't been generated for them yet. We cannot know the `startTime` of a word when it's clicked.
* **Fix**: 
  - Track the **word index** (`wIdx`) that was clicked using `pendingSeekWordIdxRef`, rather than the timestamp.
  - Issue the `seek` command to the backend `(chapter, paragraph, sentence)`.
  - When the backend finishes generating the TTS, it sends the new audio URL *along with the word timing map*.
  - Inside the `loadeddata` event of the `<audio>` element, the frontend looks up `pendingSeekWordIdxRef` in the newly arrived timing data to resolve the exact millisecond `seekTime` and snaps the audio to that position.

### 4. Backend TTS Manager Initialization
* **Gotcha**: When instantiating the TTS model via `TTSManager` for the web server, passing `None` as the console argument causes an `AttributeError` because the TTS engine attempts to call `console.print()`.
* **Fix**: Always instantiate and pass a `rich.console.Console` object to `tts_manager.create_model()`.

- The web frontend is fully wired up to the backend.
- The Bookshelf loads recent books and supports uploading.
- The Reader supports synchronized word-level highlighting, playback controls, bidirectional scrolling, and word-level seeking.
- Run `npm run build` in the `lyricflow-ebook-reader` directory whenever React changes are made.
- Run `python -m lue --web` from the project root to start the server.
